Your SQL query

Take a look at the query you just ran:


select
s.objid, s.ra, s.dec, s.u, s.g, s.r, s.i, s.z

from
star s, dbo.fGetNearbyObjEq(229.0128,-0.1082,3) n

where
s.objid = n.objid

Every SQL query has three parts. The select block tells you what quantities to return. This query returns the object ID, ra, dec, and five magnitudes.

The from block tells you which tables the query is searching. This query searches two tables. The star table contains information on stars (and only stars). The other table is generated on the fly by a function called fGetNearbyObjEq. fGetNearbyObjEq takes three inputs: the ra and dec of a point in the sky, and a radius (in arcminutes) to search around that point. The function outputs all objects in that region of the sky. This query searches 3 arcminutes around the center of Palomar 5 (229.0128,-0.1082).

The where block sets constraints on the data. For example, if you wanted to search for objects brighter than g = 18, you could say, "where g < 18". The where block of your query ensures that the data returned will be the same in both tables: star and the table generated by fGetNearbyObjEq. This constraint also ensures that only stars are returned, not galaxies.

To learn more about how to use SQL to search for data, see SkyServer's Searching for Data tutorial.
